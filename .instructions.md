# Bus Booking Platform - Development Standards

> Production-ready guidelines for seamless, smart, and clean UI/UX development across operator, conductor, and admin dashboards.

## 🎯 Project Overview

TibhukeBus is a comprehensive multi-company bus booking platform built with Next.js 15, React 18, TypeScript, Tailwind CSS 4, and Radix UI. This document establishes industry standards for UI/UX consistency, code quality, and development workflows across all dashboard interfaces.

**Target Dashboards:**
- Operator Dashboard (7 tabs: Dashboard, Trips, Bookings, Routes, Buses, Reports, Profile)
- Conductor Dashboard (5 features: Dashboard, My Trips, Payments, Reports, Profile)
- Company Admin Dashboard (Management, Analytics, Operators)

## 🧩 Component Architecture

### Core Components
- Button (variants: primary, secondary, ghost, destructive)
- Input (with validation states)
- Select/Dropdown
- Modal/Dialog
- Card
- Badge/Status indicator
- Table
- Tabs
- Toast notifications
- Skeleton loader
- Error boundary
- Empty state

### Component Rules
1. **Type Safety**: Strict TypeScript, no `any`
2. **Single Responsibility**: One component, one purpose
3. **Composition**: Use slots/children, not mega-props
4. **Accessibility**: ARIA labels, keyboard nav, screen readers
5. **Testing**: Unit + integration tests required

## 📝 Code Standards

### TypeScript
- Strict mode enabled
- Explicit return types
- No implicit `any`
- Interface for component props

### File Structure
```
component-name.tsx
- 'use client' directive
- Imports (external, context, components, types, utils)
- Type definitions
- Component function
- Export
```

### Naming
- **Files**: PascalCase (DashboardHeader.tsx)
- **Directories**: kebab-case (dashboard-header/)
- **Functions**: camelCase (handleSubmit)
- **Constants**: UPPER_SNAKE_CASE (MAX_RETRIES)
- **CSS classes**: kebab-case (active-tab)

### Comments
- Explain WHY, not WHAT
- Keep code self-documenting

## 🎯 UI/UX Patterns

### Loading States
- Use skeleton loaders for content
- Spinners only for actions
- Never show blank screens

### Error Handling
- Inline errors for forms
- Toast for API errors
- Error boundary for fatal errors
- Always provide retry option

### Empty States
- Contextual message + icon
- Suggested action (search, create, refresh)

### Modals
```
Header: Title + close button
Body:   Main content with scroll
Footer: Action buttons
Animation: 150ms fade-in
```

### Forms
- Required fields: red asterisk
- Validation: on blur + submit
- Errors: below field, inline
- Loading: button spinner + disabled

## 🧪 Testing Standards

### Unit Tests
- Component rendering
- User interactions
- State changes
- >80% coverage

### Integration Tests
- User workflows
- Context-based state
- API integration

### E2E Tests
- Full user journeys
- Cross-browser compatibility

### Accessibility Tests
- Keyboard navigation (Tab, Enter, Escape)
- ARIA labels
- Color contrast (WCAG AA: 4.5:1)

## 🚀 Development Workflow

### Before Coding
1. Create branch: `git checkout -b feat/feature-name`
2. Review relevant components/patterns
3. Understand data flow and dependencies

### While Coding
1. Follow type safety rules
2. Implement responsive design
3. Add loading + error states
4. Write tests
5. Use design tokens

### Before Commit
1. Lint: `npm run lint`
2. Tests pass: `npm run test`
3. Build succeeds: `npm run build`
4. No console errors
5. Responsive check
